# 필사 해빗 트래커 — 아키텍처 문서 (v2.1, 워커 제거 개정판)

> 최종 수정: 2026-07-12
> 상태: 대회 대응 설계 확정 (수직 슬라이스 구현 진행 중)
> v2 → v2.1 변경 요지: **Redis + BullMQ + 워커 컨테이너 제거** → NestJS 인프로세스 비동기 처리로 대체 (ADR 6.11).
> 사유: 유사도 검사를 Gemini API 직접 호출로 확정(ADR 6.3)하면서 별도 워커 프로세스의 존재 이유가 사라짐.
> 부수 변경: ADR 6.9에 Cloudflare Pages 검토 결과 추가(프로덕션 nginx 유지, Pages는 프리뷰 채널로 공존).

---

## 1. 개요

한글·영어 필사(손글씨) 이미지를 업로드하면 Gemini(멀티모달)로 텍스트를 인식·대조하고, 필사 습관을 기록·추적하는 웹 서비스.

개인 프로젝트로 시작하되 향후 확장을 전제로 하며, **소프트웨어 경진대회 출품**을 겸한다. 설계 기조:

- **"설계는 확장 가능하게, 배포는 싸게"** — 인터페이스 추상화로 인프라 교체 비용 최소화
- **"배운 것으로 설명 가능하게"** — 방송대 컴퓨터과학과 커리큘럼 과목(UNIX시스템, 정보통신망, 컴퓨터보안, 운영체제, 클라우드컴퓨팅, 데이터베이스시스템, 소프트웨어공학 등)과 구성 요소가 1:1로 대응되도록 기술 선택 (부록 A 참조)
- OCR(유사도 검사)은 Gemini로 확정 (ADR 6.3)
- GCE로 가는 건 조금 보류. 어느정도 프로젝트 얹어지면 시작.

## 3. 시스템 구성도

```
[사용자 브라우저]
   │ HTTPS
   ▼
[Cloudflare]  ← DNS + 프록시 ON (IP 은닉, DDoS 방어, WAF, 정적 자산 엣지 캐싱)
   │ Full(strict), Origin CA 인증서
   ▼
┌─ GCE VM (e2-small, 서울, 고정 IP) ─────────────────────┐
│                                                        │
│  [nginx] ── 정적: React 빌드 서빙                       │
│     │       /api/** → 프록시                            │
│     ▼                                                  │
│  [NestJS API]                                          │
│     ├─ 요청 처리 (무상태)                                │
│     └─ 인프로세스 백그라운드 잡                           │
│        (세션 status = 잡 상태: uploaded→processing→…)   │
│                                                        │
└─────┼──────────────────────────────────────────────────┘
      │
      ├─▶ Supabase (Postgres/Auth)
      ├─▶ R2 presigned URL 발급 / 이미지 로드
      └─▶ Gemini API (백그라운드 잡에서 호출, OCR·유사도)
      │
[클라이언트] ── presigned URL로 R2에 이미지 직접 업로드
```

## 4. 핵심 데이터 흐름 (필사 인증)

1. 클라이언트가 업로드 전 **리사이즈·압축** (긴 변 ~1600px, WebP, 장당 150~300KB 목표)
2. NestJS에 업로드 요청 → **R2 presigned URL** 발급 (이미지가 VM을 경유하지 않음 → 대역폭·디스크 부담 없음)
3. 클라이언트가 R2에 직접 업로드 → 완료 통지
4. NestJS가 세션 레코드 생성(`status='uploaded'`) 후 **즉시 응답** — 요청 스레드에서 Gemini를 기다리지 않는다
5. **인프로세스 백그라운드 처리기**(`@nestjs/schedule` 크론/인터벌)가 `uploaded` 세션을 원자적으로 클레임
   (`update … set status='processing' where status='uploaded'`, 다중 인스턴스 대비)
   → R2에서 이미지 로드 → **Gemini API**로 한·영 손글씨 인식 + 원문 대조·유사도 판정
6. 결과를 DB에 기록(`completed`/`failed`) → 클라이언트는 폴링 또는 Supabase Realtime으로 갱신

## 5. 설계 원칙 (확장 대비 5원칙)

1. **전부 컨테이너화** — 동일 이미지를 VM → 다른 VM/클라우드/k8s 어디로든 이동 가능
    1. 추후 확장성을 고려해 유연하게 대처할 수 있도록 컨테이너화
    2. 만약 해보고 포트폴리오 용으로 살려는 두고 싶고 비용 절감을 하고 싶다면 서버리스쪽으로 전환 가능. 구글 클라우드 런으로 전환하여 비용은 최소화 하되, 서비스는 살려둘 수 있음 (단, 인프로세스 스케줄러는 서버리스에서 동작 보장이 안 되므로 `QueueService` 교체 필요 — ADR 6.11 트레이드오프 참조)
    3. 다만 확장을 하자 한다면 유저가 얼마나 들어오느냐에 따라 변경 가능.
2. **유사도 검사는 비동기** — Gemini 호출이 수 초 걸리므로 요청 스레드에서 기다리지 않는다. 큐 인프라를 따로 두는 대신 **DB 상태 머신(`writing_sessions.status`) + 인프로세스 스케줄러**로 처리 (ADR 6.11). 스케일링 요구가 생기면 외부 큐로 전환 — seam은 `QueueService` 인터페이스로 유지.
3. **웹 서버는 무상태** — 세션은 JWT/DB, 파일은 R2. VM이 상시 가동이어도 로컬 디스크에 상태 저장 금지 (다중화 대비). 백그라운드 잡의 상태도 메모리가 아닌 DB(status 컬럼)에 있으므로 무상태 원칙과 정합.
    1. 확장 가능성 이야기가 나와서 일단은 이렇게 잡을까 하는데, 이건 나중에 '아 괜히 그랬네' 할 수도 있어요. 다만 사업 아이템으로 만든다면 이런 식으로 가는 게 좋지 않나 싶긴 합니다만, 웹서버로써의 state 장점을 버리는 거라 조금 ?가 띄워지긴 합니다.
4. **DB는 Supabase + 마이그레이션 체계** (ORM은 Prisma/TypeORM/ 기타 etc…)
5. **설정은 전부 환경변수**
6. 구현하면서 작업하고 싶다면 API 정의서는 Swagger로 대체할 수 있음.

### 인터페이스 추상화 (교체 가능 지점)

| 인터페이스 | 현재 구현체 (v2.1) | 교체 후보 | 교체 트리거 |
| --- | --- | --- | --- |
| `OcrService` | `GeminiOcrService` | `PaddleOcrService` (자체 모델) | API 비용이 자체 운영비 역전 시 |
| `QueueService` | 인프로세스 스케줄러 (DB 상태 머신) | pg-boss → Redis+BullMQ → Cloud Tasks | 재시도·동시성 제어 요구 증가, 워커 분리, 서버리스 회귀 시 |
| `StorageService` | `R2StorageService` (AWS SDK) — 슬라이스는 Supabase Storage 임시 | GCS / S3 | 필요 시 |

> v2의 "Python(FastAPI) 워커 분리 옵션"은 기본 구성에서 제외 (워커 컨테이너 자체가 없어짐).
> 대회 가점 항목이 언어 단위로 세분화된 경우에만 Gemini 호출부를 FastAPI 컨테이너로
> 분리하는 옵션을 재검토 (파이썬프로그래밍 과목 대응, 부록 A).

## 6. 주요 의사결정 기록 (ADR)

### 6.1 오라클 클라우드 → GCP

- 오라클 프리티어(ARM 4 OCPU/24GB)가 1순위였으나 가입 실패로 진행 불가 → GCP 채택.
- 기각: AWS — 2025-07-15부로 프리티어가 "최대 $200 크레딧 + 6개월" 방식으로 변경, 만료 시 계정 폐쇄 리스크. 장기 무료 운영 부적합.

### 6.2 ~~BullMQ + Redis → Cloud Tasks~~ (6.8로 대체됨)

- v1 결정: Cloud Run에서 폴링 워커가 불가하여 push 방식(Cloud Tasks) 채택.
- **상태: Superseded** — VM 전환으로 전제가 사라짐 (ADR 6.8 → 다시 6.11로 대체).

### 6.3 PaddleOCR → Gemini API (유지·확정)

- 인식 대상이 한글+영어 혼용 손글씨. PaddleOCR는 한글 손글씨 인식률·운영 부담(메모리 1~2GB, 모델 로딩) 문제.
- Gemini API로 대체 → Python OCR 워커 제거, 원문 대조·채점 기능 확장 여지.
- **2026-07-06 구현 확정**: NestJS가 Gemini를 직접 호출. 이 확정이 ADR 6.11(워커·큐 제거)의 직접 원인.
- 비용 근거: Google AI Pro 구독에 Google Developer Program 프리미엄 포함 → **월 $10 크레딧(AI Studio/Vertex 생성형 AI 전용)**. Flash 계열 기준 초기~중기 호출 커버.
    - 이건 제가 실제로 구글 드라이브 때문에 구독중인 거라 검토중인 겁니다.
- 선행 작업: 개발자 프로필에 AI Pro 연결. 배포 전 최신 가격 확인.

### 6.4 ~~Vercel → Firebase Hosting~~ (6.9로 대체됨)

- v1 결정: Vercel 이력 보존 제한 회피 + GCP 통합 관리 → Firebase Hosting.
- **상태: Superseded** — VM 전환으로 동일 서버 서빙이 유리 (ADR 6.9).

### 6.5 ~~Cloudflare 역할 축소 (DNS only)~~ (6.10으로 대체됨)

- v1 결정: Firebase/Cloud Run이 SSL·CDN·IP 보호 내장 → 프록시 OFF.
- **상태: Superseded** — 맨몸 VM 전환으로 프록시 역할 부활 (ADR 6.10).
- DNS를 Cloudflare에 남겨둔 덕에 전환 비용 0 — "도메인 관리 지점의 클라우드 비종속" 원칙의 효용 입증.

### 6.6 이미지 스토리지: Supabase Storage → Cloudflare R2 (유지)

- Supabase 무료 1GB는 압축 기준 3,000~6,000장 — 사용자 증가 시 수개월 내 소진.
- R2: 무료 10GB + **이그레스 무료**. 해빗 트래커는 과거 기록 열람이 잦아 읽기 트래픽이 지배적 → 구조적으로 유리.
- 기각: GCS — 상시 무료가 미국 리전 5GB뿐, 서울 유료 + 이그레스 과금.
- 현황: 수직 슬라이스는 Supabase Storage(presigned URL)로 임시 구동 중 → `StorageService` 교체로 R2 전환 예정.
- 후속: 원본 보존 정책(OCR 후 압축본만 유지, 장기 썸네일 교체) 수립 예정.
    - 이미지를 webp로 압축해서 저장한다거나, 하는 정책이 필요합니다.
    - 핸드폰 사진의 경우 기본 2mb에서 3mb 쯤 하기 때문에, 압축성이 가장 좋은 webp로 변환한다면 다소 용량에 대한 부담은 줄어들 거예요.

### 6.7 서버리스(Cloud Run) → GCE VM 【v2】

- **배경**:
    - ① 시연 중 콜드 스타트(수 초~수십 초)는 대회에서 치명적 리스크,
    - ② 인프라 직접 운영 경험 획득 목적,
    - ③ 대회 채점에 "학습 과목 활용" 가점 존재 — VM 직접 운영은 UNIX시스템·정보통신망·컴퓨터보안·운영체제·클라우드컴퓨팅 과목을 동시에 커버.
- **결정**: GCE e2-small(2 vCPU/2GB), 서울(asia-northeast3), Ubuntu 24.04, 고정 외부 IP. Docker Compose로 전 스택 구동. Gemini 채택 + 워커·Redis 제거(6.11)로 컨테이너가 2개뿐이라 2GB로 더욱 여유.
    - 구현은 이쪽이 빠르니 돌려보고 추후 추가 수정 등이 필요할 수는 있습니다.
- **기각 대안** (상세 비교는 부록 B):
    - Cloud Run min-instances=1 — 콜드 스타트는 해결되나 비용을 내고도 운영 경험·과목 어필이 없음. 다만 GCP 안에서 모든 걸 해결할 수는 있음.
    - GCE e2-micro 상시 무료 — 미국 리전 한정으로 레이턴시 130~150ms, RAM 1GB 제약. 시연 품질 리스크.
    - 외부 VPS(Vultr/Hetzner) — 더 저렴하나 GCP 통합 관리 포기.
    - Azure VM — 클라우드컴퓨팅 과목(Azure 기반)과 과목명 일치 최대. GCP 통합 선호로 보류하되, 채점 기준이 과목명 단위 매칭이면 재검토 여지.
- **k8s(k3s) 불채택**: 커리큘럼에 없어 가점 무관, 복잡도만 증가. 쿠버네티스는 현재 사용하기에는 돼지 목에 진주 목걸이.

### 6.8 ~~Cloud Tasks → Redis + BullMQ 회귀~~ (6.11로 대체됨)

- v2 결정: VM 상시 가동으로 폴링 워커 제약이 사라짐. "큐 시스템 직접 운영"의 대회 설명 가치를 근거로 Redis + BullMQ 채택.
- **상태: Superseded** — Gemini 직접 호출 확정(6.3)으로 워커 프로세스 자체가 사라져 전제가 소멸 (ADR 6.11).

### 6.9 Firebase Hosting → 동일 서버(nginx) 정적 서빙 【v2, 2026-07-12 재확인】

- **배경**: 서버가 상시 가동이므로 별도 호스팅의 콜드 스타트 회피 가치가 사라짐. "프론트~백엔드~인프라를 한 서버에서 직접 구성"하는 스토리가 대회에 유리.
- **결정**: nginx가 React 빌드 서빙 + `/api/**`를 NestJS로 프록시. 동일 도메인 → CORS 원천 제거.
- **트레이드오프**: 글로벌 CDN 상실 — 대회·초기 규모에서 무의미. Cloudflare 프록시의 기본 캐시가 js/css/폰트/이미지를 확장자 기준으로 엣지 캐싱하므로 대부분 보완됨(단 HTML은 기본 미캐싱 — index.html만 오리진까지 옴).
- **2026-07-12 추가 — Cloudflare Pages 검토 결과**: 팀원이 셋업한 Pages 배포(wrangler)로의 전환을 검토했으나 **프로덕션은 nginx 유지로 확정**.
    - 근거: Pages `_redirects`는 외부 도메인으로 200 프록시 불가(공식 문서 명시) → Pages로 가면 백엔드 CORS 활성화 또는 Pages Functions 프록시 글루 코드가 필요해 동일 도메인 전제(백엔드 `setGlobalPrefix('api')`)를 깸. 정적 서빙+TLS 종단은 자기 운영 어필이 가장 쉬운 부분이라 대회 스토리 측면에서도 손해.
    - 공존: 팀원의 Pages 셋업은 **프리뷰/스테이징 채널**(브랜치별 미리보기, 무료 500빌드/월)로 유지.
    - 재검토 조건: 대회 후 서버리스 회귀(부록 B ①) 시 — 그 시점엔 Cloudflare의 신규 프로젝트 방향인 Workers(정적 에셋 내장)도 함께 검토.

### 6.10 Cloudflare 프록시 ON 복원 【v2】

- **배경**: 공인 IP가 노출된 VM은 IP 은닉·DDoS 방어·WAF가 다시 필요.
- **결정**: A 레코드 → VM 고정 IP, 프록시 ON(주황 구름). SSL 모드 **Full(strict)** + **Origin CA 인증서**(15년 유효)를 nginx에 설치.
- Let's Encrypt 자동 발급은 프록시 ON 상태에서 검증이 꼬일 수 있어 Origin CA 방식 채택.

### 6.11 Redis + BullMQ → NestJS 인프로세스 비동기 【신규, v2.1】

- **배경**: 유사도 검사를 Gemini API 직접 호출로 확정(6.3, 2026-07-06)하면서 별도 워커 프로세스의 존재 이유가 사라짐. 유사도 "계산"과 streak/통계 "상태 전이·집계"가 모두 NestJS 한 프로세스 안에 있어 크로스-프로세스 조율(잡 테이블·폴링·콜백·HMAC)이 통째로 불필요.
- **결정**: Redis·워커 컨테이너 없이 **NestJS 인프로세스 백그라운드 처리**.
    - `writing_sessions.status`(`uploaded`→`processing`→`completed`/`failed`)를 잡 상태로 사용 — 별도 잡 테이블 없음.
    - `@nestjs/schedule` 크론/인터벌이 `uploaded` 세션을 **원자적으로 클레임**(`update … where status='uploaded'`) — 다중 인스턴스로 늘어나도 이중 처리 없음.
    - 비동기로 두는 이유는 유지: Gemini 호출이 수 초 걸려 요청을 붙잡지 않기 위함.
- **효과**: 컨테이너 4개(nginx/nestjs/redis/worker) → **2개(nginx/nestjs)**. e2-small 2GB 메모리 여유 확대, Redis 지속성(AOF)·포트 관리 등 운영 표면 축소.
- **트레이드오프**:
    - BullMQ가 주던 재시도·백오프·동시성 제어를 단순한 형태로 직접 구현해야 함. 요구가 커지면 **pg-boss(추가 인프라 없이 Postgres 기반) → Redis+BullMQ** 순으로 전환. seam은 `QueueService` 인터페이스로 유지.
    - 인프로세스 스케줄러는 **서버리스(Cloud Run min=0) 회귀 시 동작 보장이 안 됨**(인스턴스가 0으로 줄면 잡이 안 돎) — 회귀 시 `QueueService`를 Cloud Tasks 구현체로 교체하는 것이 전제 조건.
- **대회 어필 관점**: "큐 시스템 직접 운영"(6.8의 채택 사유) 재료는 사라지나, **관리형 큐(v1) → 자체 큐(v2) → 인프로세스+DB 상태 머신(v2.1)으로 이어진 의사결정 과정** 자체가 규모 적합성(YAGNI)·의존성 역전(seam 유지)의 소프트웨어공학 어필로 더 설득력 있음. 원자적 클레임은 데이터베이스시스템(트랜잭션·동시성 제어) 어필 재료.

- **그 프리미엄으로 사는 것**: 서울 리전 저지연(시연 품질), 결제·IAM·모니터링·스냅샷의 GCP 단일 콘솔 관리, VPC 방화벽 등 GCP 네이티브 보안 계층(어필 재료), 성장 경로(GKE/Cloud Run 회귀)와의 연속성.
- **합리성 결론**: "되도록 GCP에서 관리"라는 제약하에서 ③이 유일한 정합 해이며, 제약을 풀어도 프리미엄이 편익 대비 과하지 않음. 단 두 조건에서는 재검토 — (a) 채점 기준이 과목명 직접 매칭임이 확인되면 ⑦ Azure, (b) 예산 0원이 절대 조건이면 ④ e2-micro.
- **출구 전략**: Docker Compose 구조는 ③~⑦ 어디로든 그대로 이식 가능. 대회 종료 후 비용을 0원대로 낮추려면 스냅샷 후 인스턴스 중지, 또는 ①(서버리스)로 회귀 — `QueueService` 구현체 교체만으로 가능.
