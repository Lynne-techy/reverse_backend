설계 노트 · 데이터베이스시스템

스키마의 각 결정을 정규화·동시성 제어·접근 패턴 관점에서 설명한다.

상태 머신 = 원자적 클레임 대상 동시성 제어

writing_sessions.status의 CHECK 제약(pending·uploaded·processing·completed·failed)이 잡 상태다. 스케줄러가 update … where status='uploaded'로 원자적 클레임 → 별도 큐 인프라 없이 DB 트랜잭션만으로 다중 인스턴스 이중 처리를 차단.

관심사 분리 · 1:1 정규화

user_statistics를 users에서 떼어냈다. streak·집계는 필사마다 갱신되는 고빈도 write라, 갱신 주기가 다른 프로필과 한 행에 두면 잠금 경합·행 부풀림이 생긴다. 접근 패턴으로 테이블을 가른 결정.

복합 기본키 무결성

user_daily_activity(user_id, activity_date) — 사용자×날짜 유일. book_infos(translation_code, book_no) — 번역본별 책 유일. 자연키 복합 PK로 중복을 스키마 차원에서 원천 차단.

의도적 비정규화 읽기 최적화

writing_sessions는 key_verse_id와 별도로 book_no·chapter·start/end_verse_no를 보관한다. 필사 범위는 여러 절에 걸치고, 조회가 지배적인 해빗 트래커에서 verses 조인 없이 범위를 읽기 위한 선택.

인증 위임 · FK 체인 컴퓨터보안

모든 사용자 데이터는 users로 FK가 모이고, users.id는 다시 auth.users.id(Supabase Auth)로 FK. 인증 신원과 도메인 프로필을 1:1로 고정해 고아 레코드를 방지.

암묵 관계 (개선 여지) 미선언 FK

book_infos ↔ verses는 (translation_code, book_no)로 개념적으로 이어지나 FK가 선언돼 있지 않다. 현재는 앱 레벨 보장 — 선언적 복합 FK로 승격하면 무결성이 스키마로 강제된다.
