비동기 인증 흐름

Gemini 호출이 수 초 걸리므로 요청 스레드를 붙잡지 않는다. 세션 상태 컬럼이 곧 잡 상태이며(pending→processing→completed/failed), 완료 요청 핸들러가 원자적으로 클레임한 뒤 인프로세스 백그라운드 잡을 띄운다(폴링 스케줄러 없음). 조건부 update라 동시 요청이 겹쳐도 정확히 하나만 처리한다.

설계 변천 — 큐 의사결정 3단

현재 구조는 처음부터 이렇지 않았다. 큐 처리 방식을 세 번 바꿨고, 매번 전제가 무너져 구현 전에 이전 결정을 폐기했다(v1·v2는 ADR 단계에서 superseded — 코드로 짜지 않았다). 그래서 비동기 처리를 갈아끼울 지점을 QueueService라는 교체 경계(seam)로 문서에 명시해 둔 게 논거다. 넣은 것보다 뺀 것을 설명할 수 있다는 게 이 프로젝트의 논거다.

Cloud Tasks
ADR 6.2 · 관리형 푸시 큐
채택 이유: Cloud Run 서버리스는 폴링 워커가 불가 → 푸시 방식 큐가 필요.

컨테이너 2 (nginx/api) · 2GB 여유↑
명시된 교체 경계 · 의존성 역전
v1·v2는 구현 전 폐기됐고, 현재 코드엔 QueueService 타입 대신 HandwritingCheckService + ConcurrencyLimiter + writing_sessions 상태 전이가 그 자리를 채운다. 재확장이 필요하면 이 경계로 큐를 다시 끼운다 — 규모 적합성(YAGNI)의 실증이다.
재확장 경로 (요구 발생 시) → pg-boss · 추가 인프라 0 → Redis + BullMQ · 워커 분리 → Cloud Tasks · 서버리스 회귀
